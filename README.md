# MC 联机器（Minecraft Connector）

> 让每个 MC 玩家都能轻松联机，无需内网穿透知识，一键创建房间，好友秒进。
>
> **零服务器成本 · 纯 P2P · 连接码交换 · 开源免费**

**承诺：全开源 | 永久免费 | 无任何充值功能 | 社区驱动 | 零运营成本**

---

## 一、可行性分析

### 1.1 市场可行性：⭐⭐⭐⭐⭐（极高）

| 因素               | 分析                                                               |
| ------------------ | ------------------------------------------------------------------ |
| **用户痛点** | 非网易版 MC 玩家中，80%+ 不会配置端口转发/内网穿透，联机是刚需     |
| **竞品现状** | 樱花FRP、OpenFRP 等工具仍需一定技术知识；Radmin VPN 安装配置门槛高 |
| **目标用户** | 学生群体为主，对"一键式"工具有强烈需求                             |
| **传播性**   | MC 玩家社区黏性强，UP 主视频宣传效果显著，适合"口口相传"           |

### 1.2 技术可行性：⭐⭐⭐⭐（高）

| 挑战                     | 解决方案                            | 难度   |
| ------------------------ | ----------------------------------- | ------ |
| NAT 穿透                 | QUIC 协议 + UDP 打洞 + STUN 协调    | ⭐⭐⭐ |
| 中继转发（打洞失败兜底） | 轻量级社区中继服务器                | ⭐⭐   |
| 跨版本兼容               | 基于 Minecraft 标准协议的版本适配层 | ⭐⭐⭐ |
| MOD 自动同步             | 增量文件同步引擎，仅传输差异        | ⭐⭐   |
| 内嵌 MC 服务器           | 集成优化过的 Paper/Leaves 服务端    | ⭐⭐   |

### 1.3 运营可行性：⭐⭐⭐⭐（高）

- **免费模式可持续性**：社区捐赠支撑中继服务器（~500-1000元/月即可支撑数千用户）
- **开源模式**：降低社区贡献门槛，吸引开发者共建
- **风险点**：中继服务器成本随用户量增长，需设计 P2P 优先策略降低服务器负载

### 1.4 总体评估：**高度可行 ✅**

技术方案成熟，用户需求真实且强烈，开源免费模式适合中国市场。建议立即启动。

## 二、系统架构设计

> 🆕 **最新设计：[零服务器架构详解 →](docs/serverless-architecture.md)**
>
> 核心模式：**P2P 直连 + 连接码交换**，不需要任何中央服务器。信令服务器和中继服务器均为**可选增强组件**。

**关键区别**：
- ❌ **不再需要**信令服务器 — 房间发现通过连接码（QQ/微信发送）
- ❌ **不再需要**自建 STUN — 用 Google 免费公共 STUN
- ✅ **可选**社区中继 — 用户自己跑一个二进制文件就能当中继
- ✅ P2P 直连覆盖 ~75-85% 场景，中继仅为兜底

### 2.2 技术栈选型

| 层级                   | 技术                                | 选型理由                                 |
| ---------------------- | ----------------------------------- | ---------------------------------------- |
| **桌面框架**     | Tauri 2.x                           | 体积小（~5MB），内存占用低，适合游戏场景 |
| **GUI 前端**     | React 18 + TypeScript + TailwindCSS | 生态成熟，开发效率高                     |
| **后端语言**     | Rust                                | 与 Tauri 无缝集成，网络性能优异          |
| **QUIC 实现**    | quinn（Rust）                       | 纯 Rust 实现，异步友好，NAT 穿透支持好   |
| **STUN/TURN**    | 自研轻量实现                        | 降低外部依赖，可控性强                   |
| **MC 服务端**    | Paper / Leaves                      | 性能优化好，插件生态丰富                 |
| **信令服务器**   | Go + WebSocket                      | 高并发支持，部署简单，单二进制文件       |
| **中继服务器**   | Go + QUIC                           | 与 QUIC 客户端配合，高性能转发           |
| **MOD 差分同步** | 自研 + jdiff 启发                   | 仅传输差异文件，节省带宽                 |
| **插件系统**     | WebAssembly (wasmtime)              | 安全沙箱，多语言支持，热加载             |

### 2.3 网络架构（核心难点）

```
                    联机连接建立流程
                  
  主机 (Host)                         客户端 (Client)
     │                                     │
     │  ① 创建房间                          │
     │  ├─ 启动内嵌 MC 服务器               │
     │  ├─ 生成本地盘符 JAR                 │
     │  └─ 向信令服务器注册房间              │
     │                                     │
     │  ② 信令服务器返回房间码 (6位)         │
     │                                     │
     │                    ③ 输入房间码 ────→│
     │                    ④ 查询主机地址 ←──│
     │                                     │
     │  ⑤ NAT 类型探测                      │
     │  ←── STUN 请求 ──────────────────→  │
     │  ←── STUN 请求 ──────────────────→  │
     │                                     │
     │  ⑥ 连接策略选择                      │
     │  ┌─────────────────────────────┐    │
     │  │ 同局域网？ → 直连 (LAN)     │    │
     │  │ 锥形NAT？  → UDP打洞 (P2P)  │    │
     │  │ 对称NAT？  → 中继转发       │    │
     │  └─────────────────────────────┘    │
     │                                     │
     │  ⑦ 建立 QUIC 隧道                    │
     │  ←══════ QUIC 加密通道 ═══════→     │
     │                                     │
     │  ⑧ MOD 同步检查                      │
     │  ←── MOD 文件列表 + Hash ───────→   │
     │  ←── 增量同步缺失 MOD ──────────→   │
     │                                     │
     │  ⑨ 客户端启动 MC                     │
     │  └─ 自动配置连接到隧道本地端口        │
     │                                     │
     │  ⑩ 开始游戏！🎮                      │
     │  ←══════ Minecraft 协议 ═══════→    │
     └─────────────────────────────────────┘
```

#### 连接策略详解

| 场景               | 触发条件           | 方案          | 延迟     | 成功率 |
| ------------------ | ------------------ | ------------- | -------- | ------ |
| **LAN 直连** | 同一局域网 / VPN   | 直接 TCP      | <1ms     | 100%   |
| **UDP 打洞** | 至少一方为锥形 NAT | QUIC over UDP | 10-50ms  | ~70%   |
| **中继转发** | 双方均为对称 NAT   | 服务器中转    | 30-100ms | 100%   |

> **设计目标**：80%+ 的连接走 P2P（LAN + UDP打洞），仅 20% 走中继，大幅降低服务器成本。

## 三、核心模块设计

### 3.1 隧道引擎 (Tunnel Engine)

```
┌─────────────────────────────────────────┐
│            Tunnel Engine                 │
│                                          │
│  ┌─────────┐  ┌─────────┐  ┌──────────┐│
│  │ NAT     │  │ QUIC    │  │ TCP      ││
│  │ Detector│  │ Manager │  │ Bridge   ││
│  │         │  │         │  │          ││
│  │ STUN    │  │ 连接    │  │ 本地TCP  ││
│  │ 客户端  │  │ 管理    │  │ ←→ QUIC  ││
│  └─────────┘  └─────────┘  └──────────┘│
│                                          │
│  对外接口：                               │
│  • create_tunnel() -> TunnelHandle       │
│  • join_tunnel(room_code) -> TunnelHandle│
│  • tunnel.local_port -> u16             │
│  • tunnel.status -> ConnectionStatus    │
└─────────────────────────────────────────┘
```

**关键设计决策**：

- 使用 QUIC 作为隧道协议：天然支持 NAT 穿透，内置加密，0-RTT 握手机制
- TCP Bridge 将 Minecraft 的 TCP 流量包装成 QUIC Stream
- NAT 探测器通过多个 STUN 请求判断 NAT 类型（RFC 5780）
