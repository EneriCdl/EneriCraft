# MC 联机器 — 零服务器架构设计

## 核心理念

> **软件本身不依赖任何中央服务器。** 所有连接都是 P2P 直连。信令交换通过"连接码"由用户自己在 QQ/微信 中传递。

---

## 一、传统方案 vs 零服务器方案

| | 传统联机工具 | 我们的零服务器方案 |
|---|---|---|
| **房间发现** | 需要信令服务器中转 | 主机生成"连接码"，通过QQ/微信发给朋友 |
| **NAT探测** | 需要自建 STUN 服务 | 使用 Google 免费公共 STUN 服务器 |
| **联机通道** | 依赖中继服务器兜底 | QUIC UDP 打洞直连（覆盖70%+场景） |
| **中继兜底** | 官方服务器中转 | **社区中继**：任何人都能用自己电脑/VPS开中继，社区共享 |
| **MOD同步** | 通过服务器 | P2P 直传（走的同一个 QUIC 隧道） |
| **每月费用** | ¥68-1850 | **¥0** |

## 二、连接码机制（替代信令服务器）

### 工作流程

```
主机 (Host)                          朋友 (Client)
    │                                     │
    │ ① 点击"创建房间"                     │
    │    自动探测 NAT 类型（Google STUN）   │
    │    启动内嵌 MC 服务器                │
    │                                     │
    │ ② 生成"连接码"                       │
    │  ┌─────────────────────────┐        │
    │  │ MC-CONNECT-v1           │        │
    │  │ eyJ2ZXJzaW9uIjoxLCJ... │ ← base64 编码的连接信息
    │  │ (约 200-400 字符)       │        │
    │  └─────────────────────────┘        │
    │  也可生成 QR 码                      │
    │                                     │
    │ ③ 复制连接码 ──→ 微信/QQ 发给朋友 ──→│
    │                                     │
    │                         ④ 粘贴连接码，点击"加入" 
    │                                     │
    │ ⑤ UDP 打洞                           │
    │  ←── UDP 包互发 (基于连接码中的IP) ──→│
    │                                     │
    │ ⑥ QUIC 加密隧道建立                  │
    │  ←══════ P2P 直连 ═══════→         │
    │                                     │
    │ ⑦ MOD 差异对比 & P2P 同步            │
    │  ←── MOD 文件通过同一隧道传输 ──→    │
    │                                     │
    │ ⑧ 🎮 开始游戏！                      │
    └─────────────────────────────────────┘
```

### 连接码内容

```
连接码 = base64(压缩后的JSON)，包含：

{
  "v": 1,                          // 协议版本
  "ep": [                          // 主机端点列表
    {"ip": "123.45.67.89", "p": 54321},   // 公网地址 (STUN获取)
    {"ip": "192.168.1.5",  "p": 54321}    // 内网地址 (LAN直连用)
  ],
  "pk": "base64公钥...",           // QUIC 自签名证书公钥
  "mv": "1.21",                    // MC 版本
  "mh": "sha256-hash...",          // MOD 清单 Hash
  "ts": 1712345678                 // 时间戳 (防重放)
}
```

> 连接码约 200-400 字符，复制粘贴非常方便。也可以生成 QR 码，手机扫一下就行。

```
                      ┌─────────────┐
                      │  获取连接码  │
                      └──────┬──────┘
                             │
                      ┌──────▼──────┐
                      │  解析端点列表 │
                      └──────┬──────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
        ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼─────┐
        │ 尝试 LAN  │ │ 尝试 UDP  │ │ 尝试 TCP  │
        │  直连     │ │   打洞    │ │   打洞    │
        │ (内网IP)  │ │ (公网IP)  │ │ (公网IP)  │
        └─────┬─────┘ └─────┬─────┘ └─────┬─────┘
              │              │              │
              └──────────────┼──────────────┘
                             │
                    ┌────────▼────────┐
                    │  任一成功？      │
                    └────┬───────┬────┘
                         │       │
                    成功 │       │ 全部失败
                         │       │
               ┌─────────▼──┐ ┌──▼──────────────┐
               │ QUIC 隧道  │ │ 提示用户：        │
               │ 建立 ✅    │ │ "无法直连，建议：  │
               └────────────┘ │ ① 检查防火墙设置  │
                              │ ② 使用社区中继    │
                              │ ③ 自行部署中继    │
                              └──────────────────┘
```

### 各策略成功率

| 策略 | 适用场景 | 预计成功率 | 延迟 |
|------|----------|-----------|------|
| LAN 直连 | 同一 WiFi/局域网 | 100%（局域网内） | <1ms |
| UDP 打洞 | 至少一方锥形 NAT | ~70% | 10-50ms |
| TCP 打洞 | 某些对称 NAT | ~15%（补充） | 20-80ms |
| **综合 P2P 成功率** | | **~75-85%** | |
| 社区中继 | P2P 全部失败 | 100%（但需有人开中继） | 30-150ms |

## 四、社区中继系统（可选，P2P 失败时的兜底）

### 设计思路

> 任何人都可以运行中继服务器。只需要一台有公网 IP 的电脑/VPS，运行一个几 MB 的二进制文件。

### 使用场景

```
场景一：你有公网 IP 的 VPS
  → 跑一个 relay-server  → 在软件里填上地址 → 你和朋友都能用

场景二：你朋友有公网 IP
  → 朋友跑 relay-server → 你填上朋友的地址 → 搞定

场景三：社区热心人
  → 跑一个公共中继 → 把地址分享到群里 → 大家都能用

场景四：你和朋友网络都好
  → P2P 直连成功 → 根本不需要中继
```

### 中继服务器

```
relay-server (单文件 Go 二进制，约 8MB)
├── 零配置启动：./relay-server --port 3478
├── 不存储任何数据，纯转发
├── 内存占用：每个房间约 2-5MB
├── 带宽：取决于用户游戏行为，平均每个房间 50-200KB/s
└── 可限制：最大房间数、每房间带宽上限
```

## 五、零服务器版本 vs 官方服务器版本

我们把两种模式都做进软件里，用户自己选：

```
┌─────────────────────────────────────────────┐
│           MC 联机器 主界面                    │
│                                              │
│  ┌─────────────┐    ┌─────────────┐         │
│  │             │    │             │         │
│  │  创建房间    │    │  加入房间    │         │
│  │  (生成连接码) │    │  (输入连接码) │         │
│  │             │    │             │         │
│  └─────────────┘    └─────────────┘         │
│                                              │
│  ──────── 高级选项 ───────────────────      │
│  ☐ 使用社区中继 (P2P失败时自动切换)          │
│    中继地址: [community-relay.example.com]   │
│  ☐ 使用官方服务器 (自动发现房间、好友系统)    │
│                                              │
└─────────────────────────────────────────────┘
```

| 功能 | 零服务器模式 | 官方服务器模式（可选） |
|------|-------------|---------------------|
| 创建/加入房间 | ✅ 连接码 | ✅ 房间码 + 房间列表 |
| P2P 联机 | ✅ | ✅ |
| MOD 同步 | ✅ P2P 直传 | ✅ P2P 直传 |
| 房间自动发现 | ❌ 需手动发码 | ✅ 信令服务器 |
| 好友系统 | ❌ | ✅ |
| 在线状态 | ❌ | ✅ |
| 中继兜底 | 依赖社区中继 | 官方中继 |
| **月度费用** | **¥0** | ¥68-1850 |

## 六、对 TODO 清单的影响

零服务器方案的改动：

```
保持不变的任务：  P1-01~08 隧道引擎（STUN/UDP打洞/QUIC/TCP Bridge）
                  P1-16~22 MC服务端管理器
                  P1-31~40 前端GUI
                  P2-01~06 MOD同步
                  P3 插件系统、MOD生成器、语音

新增任务：        P1-N01 连接码生成器（JSON→base64→QR码）
                  P1-N02 连接码解析器（base64→JSON→连接策略）
                  P1-N03 剪贴板监听（自动检测连接码）

降级为可选：      P1-09~15 信令服务器 → 变为"可选增强组件"
                  P2-07~12 中继服务器 → 变为"社区自托管组件"
                  P2-13~16 好友系统 → 依赖信令服务器，变为可选

变为独立项目：    server/ 目录 → 可作为独立仓库（官方/社区服务器）
```

## 七、推荐策略

```
第一阶段（现在）：
  只做零服务器版本
  ├── 连接码交换
  ├── P2P 直连
  └── 社区中继（提供二进制，社区自己跑）

  用户体验：
  "复制这串码发给朋友 → 朋友粘贴 → 开始玩"
  就像分享网盘链接一样简单。

第二阶段（有用户量后）：
  如果有人愿意捐赠服务器 → 部署官方信令+中继
  ├── 自动房间发现
  ├── 好友系统
  └── 官方中继兜底

  用户体验升级：
  "创建房间 → 朋友在列表里看到 → 一键加入"
```

## 八、总结

| 问题 | 答案 |
|------|------|
| **能零成本吗？** | ✅ 能。连接码替代信令服务器，Google STUN 替代自建STUN |
| **P2P 覆盖多少用户？** | ~75-85%。剩下 15-25% 用社区中继 |
| **用户体验会差很多吗？** | 不会。就多一步"复制连接码发给朋友"，QQ/微信人人都有 |
| **以后能加服务器吗？** | 能。架构完全兼容，服务器是可插拔的增强组件 |
| **中继谁提供？** | 社区。一个 8MB 的二进制文件，有公网 IP 就能跑 |
